# Orchestration State Schema

Two JSON files act as the **single source of truth** for an in-progress plan execution. Whatever skill is driving the work (`hs-exec-plan`, `hs-team`, or a downstream skill resuming after interruption) reads and writes these files. The ExecPlan markdown is a *narrative* of intent; these files are the live state.

**Location:** `docs/runs/<plan-slug>/state/`

```
docs/runs/<plan-slug>/
├── state/
│   ├── features.json          (per-task lifecycle)
│   └── validation-state.json  (per-case verification)
└── runs/
    └── <timestamp>/           (artifacts written by hs-user-test)
        ├── <case-id>/
        └── synthesis.json
```

Any agent can `cat` the two files in `state/` and pick up exactly where the prior session left off — without re-reading the ExecPlan or replaying its history.

## features.json

Per-task lifecycle. Generated by `hs-planner` at plan ingestion; mutated by the controller (`hs-team` or `hs-exec-plan`) as tasks advance.

### Schema

```json
{
  "plan": "ai-daily",
  "generated_at": "2026-05-19T08:00:00Z",
  "features": [
    {
      "id": "T1",
      "title": "Build the public read API",
      "status": "completed",
      "milestone": "M1-foundation",
      "depends_on": [],
      "fulfills": ["UT-AI-DAILY-001", "UT-AI-DAILY-002"],
      "preconditions": "DB schema from T0 applied; .env contains VITE_SITE_URL",
      "expected_behavior": "GET /api/daily-reports returns paginated completed reports",
      "verification_steps": [
        "curl http://localhost:5173/api/daily-reports → 200",
        "Manual: load /admin and confirm admin path unchanged"
      ],
      "commit": "abc1234",
      "started_at": "2026-05-19T09:14:00Z",
      "completed_at": "2026-05-19T10:02:00Z"
    }
  ],
  "misc_buckets": [
    {
      "id": "misc-M1",
      "milestone": "M1-foundation",
      "items": [
        { "ref": "T7", "added_at": "2026-05-19T10:30:00Z" }
      ]
    }
  ],
  "dismissed": [
    {
      "from_task": "T3",
      "issue": "Older daily reports in legacy schema render without authors",
      "justification": "Legacy rows are already normalised by services layer; product confirmed at 2026-05-17 that pre-2025 backfill is out of scope and won't be revisited.",
      "dismissed_at": "2026-05-19T11:15:00Z"
    }
  ],
  "decisions": [
    {
      "at": "2026-05-19T09:45:00Z",
      "question": "Email subscription placeholder: write to DB or console.log only?",
      "options": ["DB stub row", "console.log", "drop subscribe form entirely"],
      "answer": "DB stub row",
      "answered_by": "human"
    }
  ]
}
```

### Field rules

| Field | Required | Notes |
|---|---|---|
| `id` | yes | Stable task ID from ExecPlan |
| `status` | yes | One of `pending`, `in-progress`, `completed`, `cancelled` |
| `milestone` | when plan uses milestones | Milestone ID; misc buckets use `misc-<milestone>` |
| `depends_on` | yes (may be `[]`) | Task IDs that must reach `completed` before this can start |
| `fulfills` | yes (may be `[]`) | User-test case IDs this task closes. **Only leaf tasks may claim cases.** Every case in `docs/user-tests/<feature>.md` must appear in exactly one task's `fulfills`. The controller fails plan ingestion if the union of all `fulfills` does not equal the full case set. |
| `preconditions` | yes | Plain-text state the system must be in before this task starts |
| `expected_behavior` | yes | What the user can do after the task lands that they couldn't before |
| `verification_steps` | yes | Numbered observable checks. Steps prefixed with `Manual:` require a human; everything else is agent-runnable |
| `commit` | when `status=completed` | The atomic commit SHA the implementer produced |
| `started_at` / `completed_at` | when applicable | ISO timestamps |

### Misc buckets

When the follow-up decision tree (see `skills/hs-followup-scope/`) routes a small, non-blocking issue to a misc bucket:

- One bucket per milestone, named `misc-<milestone>`
- **Maximum 5 items per bucket.** Once full, new items must open a real follow-up feature or get escalated.
- Items reference real task IDs (`ref: T7`); the task itself lives in the main `features` list with its own `status`.

### Dismissed items

Append-only. Each entry requires:

- `justification` of **at least 20 characters**, naming a concrete reason ("low priority" is not a valid justification)
- Either reference to an existing feature ID that absorbs the work, or an argument that the work will never need to be done

Once dismissed, an item is not silently dropped — it lives in this list forever as the audit trail.

### Decisions log

Every AskUser exchange (see `skills/hs-team/SKILL.md` §AskUser constraint) gets one row here: the question, the 1–4 options offered, the chosen answer, and timestamp. This is how negotiations enter the audit trail.

## validation-state.json

Per-case verification state. Generated by `hs-planner` (empty `pending` rows for every case in `docs/user-tests/<feature>.md`); mutated by `hs-user-test` after every probe run.

### Schema

```json
{
  "plan": "ai-daily",
  "generated_at": "2026-05-19T08:00:00Z",
  "cases": {
    "UT-AI-DAILY-001": {
      "status": "passed",
      "last_run": "2026-05-19T10:01:00Z",
      "evidence": "docs/runs/ai-daily/runs/2026-05-19-1001/UT-AI-DAILY-001/",
      "fulfilled_by_task": "T1",
      "pattern": "isolated"
    },
    "UT-AI-DAILY-005": {
      "status": "failed",
      "last_run": "2026-05-19T11:30:00Z",
      "evidence": "docs/runs/ai-daily/runs/2026-05-19-1130/UT-AI-DAILY-005/",
      "fulfilled_by_task": "T3",
      "pattern": "systemic",
      "failure_summary": "Three sibling cases on /api/daily-reports all return 500 — root cause likely in shared response serialiser, not in T3 alone."
    },
    "UT-AI-DAILY-007": {
      "status": "pending",
      "fulfilled_by_task": "T4"
    }
  }
}
```

### Field rules

| Field | Required | Notes |
|---|---|---|
| `status` | yes | `pending`, `passed`, `failed`, `blocked`, `inconclusive` |
| `last_run` | when ≠ `pending` | ISO timestamp |
| `evidence` | when ≠ `pending` | Path to the run artifacts directory (see `hs-user-test`) |
| `fulfilled_by_task` | yes | Task ID whose `fulfills` contains this case (cross-check with `features.json`) |
| `pattern` | when `status=failed` | `systemic` (this failure is shared across siblings, fix is cross-task) vs `isolated` (only this case) — drives whether the follow-up gets opened as a new feature or as a misc-bucket item |
| `failure_summary` | when `status=failed` | One-paragraph natural language; agents downstream use it to decide scope |

### Resume rule

When `hs-user-test` re-runs at a milestone gate or after a fix, it MUST:

1. Read `validation-state.json` to determine the previous status of every requested case.
2. **Skip cases whose status is `passed` and whose evidence references the current `BASE_SHA..HEAD_SHA` window.** They do not need to re-pass.
3. Re-run cases whose status is `failed`, `pending`, or `inconclusive`.
4. Re-run cases marked `blocked` only after the dependency is cleared.

This is the load-bearing rule for long runs — without it, every milestone gate re-runs the entire case set, which scales poorly past ten cases.

## Status transitions

```
features.json:
  pending  ──(controller dispatches implementer)──►  in-progress
  in-progress  ──(all gates pass)──►                  completed
  in-progress  ──(escalate=true OR round-budget)──►  pending (after rescope)
  any  ──(human cancels)──►                            cancelled

validation-state.json:
  pending  ──(probe runs)──►          passed | failed | inconclusive
  failed   ──(implementer fix + re-probe)──►  passed
  any case whose fulfilled_by_task → cancelled inherits status=blocked
```

## Operating contract

- **state files are the source of truth.** ExecPlan markdown narrates intent; if they disagree about whether T3 is done, trust the state file.
- **append-only fields stay append-only.** Never delete a `dismissed` entry or a `decisions` row. Wrong dismissals get a follow-up entry referencing the prior dismissal, not a deletion.
- **a malformed state file aborts the run.** Better to stop and ask than to silently regenerate fields that may have been hand-edited.
- **resume is `cat state/*.json` plus the user-test set.** No skill should require reading the entire ExecPlan markdown to resume.
